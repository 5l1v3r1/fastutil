Provides type-specialized hash maps with a small footprint optimized for fast access and insertion, but (relatively) slow enumeration and deletion. It is distributed under the GNU General Public Licence.

Package Specification

The name of a hash map class is

keytype 2 valuetype maptype.
Thus, a Long2IntTreeMap maps ordered long integers to integers. By "type" here we mean a capitalized primitive type, or Object.

Since there are eight primitive types in Java, we get 71 (!) map classes (some nonsensical class, such as Boolean2BooleanHashMap, are not generated).

Interfaces Implemented

All maps in fastMaps implement the Map. Thus, they can be just plugged in existing code, using the standard access (of course, any attempt to use the wrong type for keys or values will produce a ClassCastException). However, they also provide (whenever possible) many polymorphic versions of the most used methods that allow to avoid the tedious "type juggling" that is well known to Java programmers, and greatly enhance performance by avoiding creating temporary objects. In doing so, they implement a more stringent interface that extend the standard one.

This new interface adds some methods, which are rather obvious, and I suggest you to have a look at some of them. There are just a few caveat, mainly due to some arbitrary restriction of Java:

  • The versions of the get(), put() and remove() methods that return a primitive type cannot, of course, rely on returning null to denote the absence of a certain pair. Rather, they return a default return value, which is set to 0 cast to the return type(or false for booleans) at creation , but can be changed using the setDefRetValue() method. Note that changing the default return value does not change anything about the data structure; it is just a way to return a reasonably meaningful result, and it can be changed at any time. (However, you cannot change the default return value of a submap.)
  • For all maps that have object as keys, the get() and remove() methods do not admit polymorphic versions, as Java does not allow return-value polymorphism. Rather, the extended interfaces introduce new methods of the form getvaluetype() and removevaluetype(). Similar problems occur with first(), last() and so on.

Performance

The main reason behind fastMaps is performance, both in time and in space. The relevant methods are something like 2 to 10 times faster than those of the standard classes. The memory footprint for a table of size n is exactly the memory required for the related types times n bytes, plus a overhead of n bytes to store the state of each entry. The absence of wrappers around primitive types can reduce space occupancy by several times (this applies even more to serialized data, e.g., when you save such a data structure in a file). These data can greatly vary with your virtual machine, JVM versions, CPU etc.

To minimize the memory footprint, the classes to not keep any additional information about elements. In particular, this means that deletions are made simply by tagging suitably an entry, and that enumerations are always linear in the size of the table (rather than in the number of entries).

Note that since deletions are handled simply by tagging, they tend to slow down access.

Robustness

The code in fastMaps has passed intensive regression tests. You can trust it must in the same way you trust the standard classes.
